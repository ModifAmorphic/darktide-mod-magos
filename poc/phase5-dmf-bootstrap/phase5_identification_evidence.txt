========================================================================
DISCOVERY METHODOLOGY
========================================================================

Same source-pattern matching against LuaJIT 2.1 `lj_api.c` used in
Phase 4 (lua_pcall) and Phase 5 Step 3 (lua_pushcclosure / lua_setfield).

For each function:
  1. Read the source body in `lj_api.c` to identify distinctive
     compiled-shape features (specific constants, call targets, memory
     offsets, type-tag values).
  2. Search the binary's LuaJIT API cluster [0xc73000, 0xc90000) for
     those features via a custom capstone-based tool.
  3. Narrow by structural features (prologue arg count, body size, call
     count) until unique.
  4. Add a `disasm_check_*` matcher that asserts the feature set at the
     chosen RVA. Each matcher is also tested in the negative (must reject
     wrong RVAs).

  LUA_API int lua_type(lua_State *L, int idx)
  {
    cTValue *o = index2adr(L, idx);
    if (tvisnumber(o)) {
      return LUA_TNUMBER;
    } else if (tvislightud(o)) {       // LJ_64 && !LJ_GC64
      return LUA_TLIGHTUSERDATA;
    } else if (o == niltv(L)) {
      return LUA_TNONE;
    } else {  /* Magic internal/external tag conversion. ORDER LJ_T */
      uint32_t t = ~itype(o);
      int tt = (int)(((t < 8 ? 0x98042110u : 0x75a06u) >> 4*(t&7)) & 15u);
      return tt;
    }
  }
